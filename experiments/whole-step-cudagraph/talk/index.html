02 · The idea

One graph for the whole step

Warm up on a side stream, record the entire fwd+bwd kernel stream once, then replay it every step. Hundreds of per-kernel cudaLaunchKernel calls collapse to a single cudaGraphLaunch — the CPU stops gating the GPU. The optimizer step stays outside the graph.

03 · Why it was thought impossible — and isn't

Zero graph breaks under FSDP2

Per-layer torch.compile before fully_shard, then shard the compiled wrapper, not the inner layer. FSDP's all-gather / reduce-scatter run as eager hooks outside the compiled regions — so nothing splits the graph, and the collectives are still capturable in the recorded stream. The “≈40 breaks” become 0.

05 · The obvious objection

Reserved, not live

Capture’s ongoing footprint matches eager: live memory is unchanged. The only elevated number is a one-time setup spike while the warm-up pool and the graph’s private pool briefly coexist — and expandable_segments reclaims most of it.

root cause + the fix at 8B scale
  • The scary reserved numbers were never extra live memory — replay reuses the pinned graph pool and allocates ~nothing. The overhead is reserved slack: a private CUDA-graph pool that can’t share free blocks with the default pool.
  • PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True lets the private pool reuse freed physical pages → steady reserved ≈ eager. Works with capture; loss bit-identical.
  • At 8B (where the slack is largest) it cut the setup spike 81 → 71 GB, and steady reserved was 44.5 vs eager 44.1 GB. reshard_after_forward=True trims the spike further (unsharded params no longer resident).
  • No effect: empty_cache (before or after). Red herring: gradient checkpointing (activations aren’t the dominant term at this shape).
06 · Where it pays
Capture pays exactly when the step is launch-bound.

The megakernel collapses launch overhead. Where there is none — a large, compute-saturated step — there is nothing to collapse.
